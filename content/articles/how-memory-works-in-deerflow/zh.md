目前 GitHub 上排名第一的仓库是一个超级 agent 工具链——ByteDance 的 DeerFlow。它用 JSON 存储记忆。以下是它的工作原理。

大多数框架通过重放聊天历史来模拟记忆，把所有东西都塞进上下文窗口，希望模型挑出重要的内容。

DeerFlow 做的事情不同。

它不存储对话。它提取关于用户的事实，按置信度打分，并将适合 2,000 token 预算的内容异步注入每个提示，不会撑大上下文。

在这篇文章中，我将根据对我运行的 Docker 自托管 DeerFlow 实例进行逆向工程并检查其 JSON 记忆来分解它的工作原理。

## 什么是 DeerFlow

DeerFlow（深度探索与高效研究流）是一个开源超级 agent 工具链，通过可扩展的 skills 来编排子 agent、记忆和沙箱，几乎可以完成任何事情。

它们目前 GitHub 趋势排名第一，拥有 49.2K 星。在[这里](https://github.com/bytedance/deer-flow)查看它们的仓库。

DeerFlow UI 看起来像 ChatGPT 或 Claude 界面，左侧边栏有聊天界面和 agent 选项。

我在 MacBook 的 Docker 容器中本地运行它，通过对话测试了大约 3 小时。

我测试的内容：

- 就技术话题与 DeerFlow 对话；记忆基准测试（LOCOMO、LongMemEval）、理解 DeerFlow 架构，以及我的爱好（滑翔伞推荐）等一般话题。
- 我通过提问和让 agent 省略某些记忆来测试记忆检索。
- 记忆层是关注度不够的部分。

## 记忆界面

记忆面板将所有内容组织为：

- 用户上下文（工作 + 个人）
- 当前焦点（心头事）
- 历史层（最近几个月、更早的上下文、长期背景）
- 一个事实表

你可以看到系统了解你的所有内容、它何时学的以及它的置信度如何。

每个事实的 source 字段是实际的 thread UUID，所以系统确切地知道每条知识来自哪个对话。

## 记忆存储在哪里

DeerFlow 中的记忆以文件形式存在于磁盘上：backend/.deer-flow/memory.json。

它是本地的，跨每个会话持久化。一个结构化的 JSON 文件。

在一个会话中，DeerFlow 构建一个结构化的画像。

每个事实的内容类似于我们在记忆面板 UI 中看到的。

置信度低于 0.7 的事实不会被包含。存储上限为 100 个事实，当溢出时，首先驱逐置信度最低的。

## 它实际上是如何工作的

DeerFlow 的主 Agent 运行在 LangGraph 上，这是一个基于图的编排框架，每个 agent 回合都是有状态执行图中的一个节点。

记忆不是插件或侧车。它直接烘焙到中间件链的第 8 位，这意味着它在每个 agent 回合上自动运行。

关键组件是 MemoryMiddleware。

MemoryMiddleware 正好在 TitleMiddleware 之后、视觉和循环检测层之前。

这是刻意的，记忆更新应该在标题生成之后发生（这样 LLM 知道对话是关于什么的），但在任何循环或澄清检查缩短会话之前。

MemoryMiddleware 不同步更新记忆。它将对话排队等待异步处理。

流程是这样的：

1. 用户发送消息。Agent 响应。
2. MemoryMiddleware 过滤交换，只有用户输入和最终 AI 响应被考虑。
3. 对话被添加到带有 30 秒防抖计时器的异步队列。如果 30 秒内有任何线程的另条消息到达，计时器重置。
4. LLM 提取器针对对话运行并产生差异：要添加的新事实、要删除的事实，以及每个摘要部分的 shouldUpdate 标志。
5. 记忆 JSON 被更新。

在追加新事实之前，系统检查完全相同的内容重复（通过去除空格标准化）。

注意：相同语义事实的不同表述仍然会被添加，去重是基于文本的，不是语义的。

你可以将 model_name 指向一个更便宜的模型用于提取步骤，记忆 LLM 不需要是你最好的模型，它只需要擅长结构化提取即可。

我在步骤 3 中发现有趣的是，如果同一个 thread_id 在队列中已经有待处理的更新，新条目会替换它而不是追加。你永远不会处理过时的会话中快照，只有每个线程的最终状态被处理。

记忆摄取不会从延迟角度影响你从 agent 获取回复的用户体验。如果用户发送消息，他们会立即收到回复。

这是 agentic 系统中良好工具链的一个例子。

## 记忆实际出现在哪里

当你开始新对话时，每个适合 2,000 token 预算的事实（按置信度排序），加上所有用户、历史和 topOfMind 摘要都被注入到系统提示中，放在 `<memory>` 标签内。

没有硬编码的"前 N 条"限制；这是 token 预算，不是计数。Tiktoken 精确计数 tokens，因为事实一个接一个添加直到预算用完。

agent 在每个会话开始时看到这样的内容，来自我的实际会话，完全按照 format_memory_for_injection() 的格式。

agent 从不需要问"你在做什么？"它已经基于我的心头事历史知道了。

## 记忆检索测试

我尝试了更多测试以了解它是否能够根据我与它的对话检索我的信息。

检索按预期工作。

我尝试通过聊天删除记忆。尽管 LLM 提取器步骤显示它有 factsToRemove 差异记录，但它只是在事实表中添加了更多偏好"我不想谈论这个话题"。

肯定有一个变通方法，即修改代码库中的 json。这显示了记忆系统的局限性。

我遇到的另一个限制是，当我在一个会话中谈论记忆基准测试时，尽管我的用户配置文件显示英文偏好，但它以中文显示问题推荐。

## 结论

DeerFlow 记忆的亮点

1. 本地和你的。记忆存储没有供应商锁定。它是一个 JSON 文件。你可以读取、编辑、重置它。
2. 默认异步。记忆更新永远不会减慢你的回复速度。30 秒防抖很聪明，它吸收了会话噪音，而不会在每条消息上消耗 LLM 调用。
3. 置信度打分存储。一次对某事的提及（"我可能有一天会试试 Rust"）不会永久改变你的记忆画像。置信度阈值在弱信号积累之前过滤掉它们。
4. 原子写入。写入然后重命名是这里的正确模式。进程崩溃时的记忆损坏是一个真实的故障模式，大多数玩具实现忽略了这一点。
5. 去重。追加前的重复检查意味着系统不会以 50 种不同表述将相同的偏好分层 50 次。

## 它不做什么

1. 没有语义搜索。注入是适合 token 预算的最高置信度事实，而不是你当前查询最相关的事实。如果你存储了 100 个事实并问一个高度具体的问题，你得到的是按置信度排列的前几条事实，而不是按与该问题的相关性。
2. 没有语义去重。重复检查是基于文本的，它去除空格并精确比较字符串。我在我的实际记忆文件中看到了这一点：关于将 DeerFlow 记忆与文件系统 + markdown 方法进行比较的事实，相同的语义意图，不同的表述，存储为两个单独的条目。在一次会话后 9 个事实，这不是问题。一个月后 80 个事实，它就开始重要了。
3. 有限的智能层 + 潜在的压缩问题。记忆上限为 100 个事实，这意味着随着上下文增长它可能会吃力。虽然置信度分数有助于优先保留或省略哪些记忆。
4. 没有向量嵌入。这是一个刻意的权衡，没有嵌入流水线，没有相似性搜索。它使系统简单且依赖少，但也带来了局限性。

## 为什么它重要

大多数 agent 框架将记忆视为检索问题：
嵌入一切，存储在向量数据库中，在查询时检索最相似的块。

这有效，但有代价。
你在每个请求中添加了嵌入模型、向量存储和检索步骤。这意味着更多基础设施、更多延迟和更多复杂性。

DeerFlow 采取了不同的方法：不要存储对话，存储理解。

它运行一个异步 LLM 提取过程，将原始对话提取为结构化事实，然后将那些事实直接注入未来的提示中。

昂贵的步骤发生在回复已经交付之后，在一个防抖 30 秒的计时器上，使用你选择的任何模型。

总体而言，它兑现了作为超级 Agent 工具链的承诺。

记忆系统在当前形式下高效且生产就绪，有一些明确的局限性。

它结合了评分和注入事实的 agentic 方法，但在某些用例上仍然缺乏健壮的记忆智能层。

总体而言，作为超级 Agent 工具链深度研究和 agentic 工具，Deerflow 团队在实现这个版本的记忆系统方面做得很好。

结果是一个了解你的 agent，不是因为它记住你说的每一句话，而是因为它构建了一个关于你的偏好、目标和上下文的持久模型。而且那个模型作为一个简单的 JSON 文件存在于你的机器上。

DeerFlow 将记忆直接构建到其中间件中，所以它开箱即用，在每个对话上，默认情况下工作。

这就是值得关注的的部分。

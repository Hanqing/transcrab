这个基准测试评估各种常用大型语言模型（LLM）解决真实 Android 开发问题的能力。

## 目标

AI 辅助软件工程出现了几个衡量 LLM 能力的基准测试。Android 开发者面临现有基准测试未涵盖的特定挑战，因此我们创建了一个专注于 Android 开发的基准测试。我们发布这个基准测试的目标是：

1. 鼓励 LLM 在 Android 开发方面的改进
2. 赋能 Android 开发者通过一系列有用的模型进行 AI 辅助来提高生产力
3. 在整个 Android 生态系统带来更高质量的应用程序

我们创建了一个与模型无关的基准测试，以准确评估 LLM 在各种 Android 开发任务上的表现。

## 我们的方法论

Android Bench 通过向 LLM 呈现来自开源软件项目的真实问题和拉取请求，评估它们生成代码解决问题的能力。这种方法旨在确保任务代表开发者日常面临的挑战。

为了建立性能基线并有一个恒定的比较点，我们将 Gemini 2.5 Flash 作为基础模型。通过这样做，我们为评估中的所有模型设定了最低标准。

### 防止数据污染的保障措施

虽然采购真实世界的仓库对基准测试的有用性至关重要，但我们承认这种暴露可能导致数据污染。在当前版本的基准测试中，我们实施了以下保障措施：

- **金丝雀字符串**：我们包含标准的 BIG-BENCH 金丝雀字符串，以阻止这些任务被包含在未来的训练中。
- **轨迹验证**：我们的团队对代理工作流程进行了彻底审查，以确保成功不是奖励破解或测试不明确的结果。

### 仓库标准

我们通过应用以下标准选择仓库和单个拉取请求：

- 仓库需要包含 Android 应用或库代码，并且在 Android 开发生态系统中很受欢迎，GitHub star 数大于或等于 500。
- 每个拉取请求需要已合并、修复了报告的问题，并具有单元测试或仪表化测试等验证。

### 定义 Android 特定挑战

为了保证 Android Bench 是 Android 工程能力的衡量标准，我们策划了与 Android 开发最佳实践密切相关的任务。我们对这些任务进行分类，以确保我们符合那些约定的标准。

我们优先考虑的领域代表了构建可扩展 Android 应用程序的约定标准，例如：
- Jetpack Compose 用于 UI
- Coroutines 和 Flows 用于异步编程
- Room 用于持久化
- Hilt 用于依赖注入

我们还关注开发者经常寻求帮助的领域，例如导航迁移、Gradle/构建配置或 SDK 更新中的破坏性变化处理。

其他关注点包括核心体验，如系统 UI、相机或媒体，以及平台特定功能，如处理配置变化、可折叠设备适配和细粒度运行时权限。

## Android Bench 组成

Android Bench 包含 100 个任务，从 38,989 个拉取请求池中精选而出。完整数据集可在 GitHub 上探索。

我们分析了 GitHub 仓库和拉取请求，发现 71% 使用 Kotlin，25% 使用 Java，确认了向新编程语言标准的转变。这种转变也反映在 UI 领域——41% 使用 Jetpack Compose，同时保持 59% 基于 View 的任务，以反映许多现有代码库的现实。

GitHub 上的大多数 Android 仓库是应用程序（63%），而 Android Bench 偏向库（58%）。由于这种转变，我们可以针对更严格的限制、模块化和架构模式来测试 LLM。

为了确保任务复杂性的更平衡分布，我们还根据更改的代码行数进行了细分。近一半（46%）是 27 行以下的小更改，其次是 33% 在 27 到 136 行之间，其余 21% 超过 136 行。

在整个基准测试中，中位任务大小是 32 行更改，最大单个更改达到 435 行。

## Android Bench 测试工具

为了便于执行基准测试，Android Bench 使用了修改版的 SWE Bench 测试工具。它分为两部分：推理代理和补丁验证器。

1. 推理代理将 mini SWE 代理与可以构建和运行 Android 项目的自定义 Docker 镜像以及 Android 开发的基础提示结合起来。运行代理会产生补丁文件，然后传递给补丁验证器。

2. 补丁验证器获取这些补丁，将它们添加到代码库中，并执行测试套件。然后输出为执行分配分数所需的数据。

测试工具在我们的 GitHub 组织中可用，允许任何人独立重新创建和验证基准测试结果，或使用测试工具执行自己的任务。

## 任务来源和过滤

任务来自两个管道。第一个管道自动化查找可能成为任务的拉取请求。我们过滤所有拉取请求以确保符合以下标准：

- 来自流行的 Android 仓库——即在 GitHub 上至少有 500 star
- 包含测试
- 标记为修复仓库中的问题
- 仅包含过去 3 年的更改

这些过滤器将拉取请求集减少到任务候选对象。任务候选对象经过两轮人工审查。

第一轮审查是为了质量保证。审查者检查拉取请求以确保代码在 LLM 生成的补丁前后都能编译，描述中有足够的上下文，且不包含描述中未提及的更改。他们还为任务分配了"难度"评级，估计如果没有 LLM 辅助编写任务代码需要多长时间。

主题专家审查通过质量保证的任务，以确保每个任务足够复杂且与 Android 开发相关。

由于拉取请求的过滤，一些好的潜在候选对象在人工审查之前被排除。这是因为它们缺少测试或相关问题。在这种情况下，我们为拉取请求创建提示或添加测试，允许将额外任务包含在最终评估中。丰富后，任务经过相同的主题专家审查过程。

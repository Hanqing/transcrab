Temporal 是 JavaScript 九年努力的结晶，它将彻底改变我们处理日期和时间的方式。

## JavaScript 是如何改变的？

JavaScript 独特之处在于它运行在所有浏览器中。没有单一的"所有者"，所以你不能孤立地做出更改就期望它到处生效。你需要各方的认同。进化通过 TC39 进行，这是负责 ECMAScript 的技术委员会。

提案经历一系列成熟阶段：
- Stage 0 - 创意
- Stage 1 - 问题空间被接受
- Stage 2 - 选定了草案设计，但工作继续进行
- Stage 2.7 - 提案原则上获批，等待测试和反馈
- Stage 3 - 实现和反馈
- Stage 4 - 标准化

2018 年我第一次研究 Temporal 时，它处于 Stage 1。TC39 委员会确信问题是真实存在的。这是一个激进的提案，要为 JavaScript 带来一个全新的日期和时间库。

## 时代的产物

1995 年，Brendan Eich 被要求用 10 天冲刺创建 Mocha（后来成为 JavaScript）。在巨大的时间压力下，许多设计决策是务实的。其中之一是直接移植 Java 的 Date 实现。

在当时这很有意义。Java 处于上升期，JavaScript 被定位为其轻量级伴侣。内部甚至将这种哲学称为 MILLJ：让它看起来像 Java。

在那一刻，与 Java 的一致性比从根本上重新思考时间模型更重要。这是一个务实的权衡。网络还很年轻，大多数使用 JavaScript 的应用程序起初都很简单。

## 网络成长了，Date 没有

到 2010 年代，JavaScript 为银行系统、交易终端、协作工具和其他复杂系统提供动力，这些系统运行在地球上的每个时区。Date 越来越成为开发者的痛点。

### 可变性

开发者经常编写辅助函数，却在无意中改变了原始 Date 对象，而不是返回一个新对象。

### 不一致的月份计算

有时人们想要获取月底的最后一天，却掉进这样的陷阱：月份加一，但天数保持不变。Date 不会将无效的日历结果约束回有效日期，而是悄悄地将溢出滚动到下一个月。

### 模糊的解析

在这个例子中，字符串类似于但不完全等同于 ISO 8601。历史上，浏览器对"几乎 ISO"字符串的行为是由规范定义的。有些将其视为本地时间，有些视为 UTC，还有一个直接抛出无效输入错误。

## 库的时代

网络生态系统别无选择，只能用库来修补 Date 的缺点。如今，它们每周的下载量总计超过 1 亿次。

领军的是 Moment.js，它拥有表达性 API、强大的解析能力，以及急需的不可变性。创建于 2011 年，它迅速成为处理 JavaScript 日期和时间操作的既定标准。

Moment.js 的广泛采用带来了自己的问题。添加库意味着增加打包大小，因为它需要附带自己的 locale 信息和时区数据库中的时区数据。

## Temporal 冠军们

Stage 1 是一个重要的里程碑，但离完成还很遥远。在彭博社，这种疼痛不是理论上的。

我们在终端上大规模运行 JavaScript，使用 Chromium、Node.js 和 SpiderMonkey 等底层运行时和引擎。我们的用户，以及他们投资的金融市场，遍布地球上的每个时区。

此外，我们还有内置 Date 模型根本没有设计到的需求：
- 用户配置的时区（不是机器的时区，可以根据请求更改）
- 由 IANA 时区数据库（tzdata）更新驱动的正确历史时区行为
- 更高精度的时间戳（纳秒），而不需要永久粘合额外字段

随着时间的推移，彭博社与 Igalia 合作（包括通过持续资金支持），并贡献工程时间直接推动 Temporal 前进，直到它最终成为整个生态系统可以发布的东西。

## Temporal 今天是什么样子

Temporal 是一个顶级命名空间对象（类似于 Math 或 Intl），存在于全局范围内。下面是 Temporal 提供的类型：

### Temporal.ZonedDateTime

如果你不知道需要哪种 Temporal 类型，从 Temporal.ZonedDateTime 开始。它是没有"陷阱"的 Date 的最接近的概念替代品。

Date 代表：
- 时间的精确时刻（内部是自 epoch 以来的毫秒）
- 通过机器当前时区解释
- 具有隐式、可变的行为

Temporal.ZonedDateTime 代表：
- 时间的精确时刻
- 具有明确的时区
- 具有明确的日历
- 完整的夏令时正确性
- 全部作为不可变值

### Temporal.Instant

Temporal.Instant 是时间的精确时刻，它没有时区、没有夏令时、没有日历。它表示自 1970 年 1 月 1 日午夜以来经过的时间（Unix epoch）。与 Date 不同，Instant 以纳秒而不是毫秒度量。

### Temporal.PlainDate, PlainTime, PlainDateTime

我们还有一组"纯"类型。这些是我们所说的"挂钟时间"，因为想象墙上的 analog 时钟，它不检查夏令时或时区。

### 日历

Temporal 支持日历。浏览器和运行时附带一组内置日历，让您可以用用户首选的日历系统表示、显示和进行算术运算，而不仅仅是格式化不同的公历日期。

## 实现

实现 Temporal 是一个挑战。这是一个非常大的提案，为 JavaScript 带来的变化比编程语言历史上任何其他提案都多。

一些具体挑战包括：
- 这个巨大的规范比整个 ECMA-402（国际化规范）更大
- 规范的波动性创造了一个移动靶
- 浏览器要求几乎所有方面都高效和高性能

Temporal 是自 ES2015 以来 ECMAScript 最大的添加物。Temporal 现在有约 4500 个测试。

## 已发布和标准化

今天，Temporal 在 TC39 分级流程中达到了 Stage 4，这意味着它将成为下一个年度 ECMAScript 规范（ES2026）的一部分。但是你不需要等到那时——今天就可以使用它！

Temporal 已经在以下环境中得到支持：
- Firefox v139（自 2025 年 5 月起）
- Chrome v144（自 2026 年 1 月起）
- Edge v144（自 2026 年 1 月起）
- TypeScript 6.0 Beta（自 2026 年 2 月起）
- Safari（技术预览版中的部分支持）

## JavaScript 的更好时光

Temporal 是近十年跨公司、引擎和个人努力的结果。它代表了：
- 在 TC39 内部多年基于生态系统先前经验的共识建设
- 跨多个 JavaScript 引擎的实现工作
- Microsoft、Google、Mozilla、彭博社、Igalia、Boa 和许多独立贡献者之间的协作

我们为多年资助和支持 Igalia 在 Temporal 上的工作感到自豪。这种投资，结合开放协作，成功地将提案从创意推动到规范再到发布的现实。

Temporal 不仅仅是一个更好的 API。它证明了 JavaScript 社区可以共同解决长期存在的问题。

近 30 年后，JavaScript 终于有了一个现代的日期时间 API。

这次，我们做对了。

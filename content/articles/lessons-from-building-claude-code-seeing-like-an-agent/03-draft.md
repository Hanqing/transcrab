# 构建 Claude Code 的经验教训：像 Agent 一样看世界

构建 agent 工具的最难部分之一是构建它的行动空间。

Claude 通过工具调用来行动，但在 Claude API 中有多种方式可以用 bash、skills 以及最近的代码执行等原语来构建工具（在 [@RLanceMartin 的新文章](https://x.com/RLanceMartin/status/2027450018513490419)中阅读更多关于 Claude API 程序化工具调用的内容）。

面对这些选项，你如何设计 agent 的工具？你只需要一个工具，比如代码执行或 bash？如果你有 50 个工具，每个对应 agent 可能遇到的一个用例呢？

为了让自己进入模型的思维，我喜欢想象被给了一道难数学题。你想要什么工具来解决它？这取决于你自己的技能！

纸是最低限度，但你会受到手动计算的限制。计算器会更好，但你需要知道如何操作更高级的选项。最快最强大的选项是电脑，但你必须知道如何用它来编写和执行代码。

这是设计 agent 的一个有用框架。你想给它形状符合其自身能力的工具。但你怎么知道那些能力是什么？你要注意、阅读它的输出、做实验。你学会像 agent 一样看世界。

以下是我们在构建 Claude Code 时通过关注 Claude 学到的一些经验。

## 改进引导与 AskUserQuestion 工具

在构建 AskUserQuestion 工具时，我们的目标是提高 Claude 提问的能力（通常称为引导）。

虽然 Claude 可以用纯文本提问，但我们发现回答这些问题感觉花了不必要的时间。我们如何降低这种摩擦并增加用户与 Claude 之间的通信带宽？

### 尝试 #1 - 编辑 ExitPlanTool

我们首先尝试的是在 ExitPlanTool 中添加一个参数，在计划旁边有一个问题数组。这是最容易实现的，但它让 Claude 感到困惑，因为我们同时要求一个计划和一组关于计划的问题。如果用户的回答与计划所说的相冲突怎么办？Claude 需要调用 ExitPlanTool 两次吗？我们需要另一种方法。

（你可以在[我们关于提示缓存的文章](https://x.com/trq212/status/2024574133011673516)中阅读更多关于我们为什么制作 ExitPlanTool 的内容）

### 尝试 #2 - 更改输出格式

接下来我们尝试修改 Claude 的输出指令，提供一种稍微修改过的 markdown 格式，它可以用来提问。例如，我们可以要求它输出一个带有括号中替代选项的要点问题列表。然后我们可以解析并将该问题格式化为用户的 UI。

虽然这是我们可以做的最通用的更改，Claude 甚至似乎能够输出这个，但这并不能保证。Claude 会附加额外的句子、省略选项，或者完全使用不同的格式。

### 尝试 #3 - AskUserQuestion 工具

最终，我们决定创建一个 Claude 可以在任何时候调用的工具，但特别提示它在计划模式下这样做。当工具触发时，我们会显示一个模态框来展示问题，并阻塞 agent 的循环直到用户回答。

这个工具允许我们提示 Claude 进行结构化输出，并帮助我们确保 Claude 给用户多个选项。它还给了用户组合这个功能的方式，例如在 Agent SDK 中调用它或在 skills 中引用它。

最重要的是，Claude 似乎喜欢调用这个工具，我们发现它的输出效果很好。即使是设计最好的工具，如果 Claude 不理解如何调用它也不会起作用。

这是 Claude Code 中引导的最终形式吗？我们不确定。正如你在下一个例子中看到的，对一个模型有效的东西对另一个模型可能不是最好的。

## 随能力更新 - 任务与待办事项

当我们第一次推出 Claude Code 时，我们意识到模型需要一个待办事项列表来保持它在正轨上。待办事项可以在开始时写好，随着模型工作而勾选。为此我们给了 Claude TodoWrite 工具，它可以写入或更新待办事项并显示给用户。

但即便如此，我们经常看到 Claude 忘记它必须做什么。为了适应，我们每 5 轮插入系统提醒，提醒 Claude 它的目标。

但随着模型改进，它们不仅不需要被提醒待办事项列表，而且可能会发现它有限制。被发送待办事项列表的提醒让 Claude 认为它必须坚持列表而不是修改它。我们还看到 Opus 4.5 在使用子 agent 方面也变得更好，但子 agent 如何在共享的待办事项列表上协调？

看到这一点，我们用任务工具替换了 TodoWrite（[在这里阅读更多关于任务的内容](https://x.com/trq212/status/2014480496013803643)）。待办事项是关于让模型保持在正轨上，而任务更多是关于帮助 agent 相互通信。任务可以包含依赖关系，在子 agent 之间共享更新，模型可以修改和删除它们。

随着模型能力的提升，你的模型曾经需要的工具现在可能在限制它们。不断重新审视关于需要什么工具的先前假设很重要。这也是为什么坚持支持一小组具有相当相似能力配置的模型很有用。

## 设计搜索界面

对 Claude 来说特别重要的一组工具是可以用来构建自身上下文的搜索工具。

当 Claude Code 刚推出时，我们使用 RAG 向量数据库为 Claude 找到上下文。虽然 RAG 强大且快速，但它需要索引和设置，并且在各种不同环境中可能很脆弱。更重要的是，Claude 是被给予这个上下文，而不是自己找到上下文。

但如果 Claude 可以在网上搜索，为什么不搜索你的代码库呢？通过给 Claude 一个 Grep 工具，我们可以让它搜索文件并自己构建上下文。

这是我们看到的一个模式，随着 Claude 变得更聪明，如果给它正确的工具，它在构建自身上下文方面变得越来越好。

当我们引入 Agent Skills 时，我们正式化了渐进式披露的概念，允许 agent 通过探索逐步发现相关上下文。

Claude 可以读取 skill 文件，这些文件可以引用模型可以递归读取的其他文件。事实上，skills 的一个常见用途是为 Claude 添加更多搜索能力，比如给它关于如何使用 API 或查询数据库的指令。

在一年的时间里，Claude 从几乎无法构建自身上下文，到能够在几层文件中进行嵌套搜索以找到它需要的确切上下文。

渐进式披露现在是我们用来在不添加工具的情况下添加新功能的常见技术。

## 渐进式披露 - Claude Code 指南 Agent

Claude Code 目前有约 20 个工具，我们不断问自己是否需要所有这些工具。添加新工具的门槛很高，因为这给模型多了一个需要考虑的选项。

例如，我们注意到 Claude 对如何使用 Claude Code 了解不够。如果你问它如何添加 MCP 或斜杠命令是什么，它无法回答。

我们可以把所有这些信息放在系统提示中，但鉴于用户很少问这些，这会增加上下文腐烂并干扰 Claude Code 的主要工作：编写代码。

相反，我们尝试了一种渐进式披露的形式。我们给了 Claude 一个指向其文档的链接，它可以加载以搜索更多信息。这有效，但我们发现 Claude 会将大量结果加载到上下文中以找到正确答案，而实际上你只需要答案。

所以我们构建了 Claude Code 指南子 agent，当你询问关于它自身的问题时，Claude 被提示调用它，子 agent 有关于如何很好地搜索文档以及返回什么的详细指令。

虽然这并不完美，当你问它如何设置自身时 Claude 仍然可能感到困惑，但比以前好多了！我们能够在不添加工具的情况下向 Claude 的行动空间添加内容。

## 艺术，而非科学

如果你希望得到一套关于如何构建工具的严格规则，遗憾的是这不是这份指南。为你的模型设计工具既是艺术也是科学。它在很大程度上取决于你使用的模型、agent 的目标以及它运行的环境。

经常实验，阅读你的输出，尝试新事物。像 agent 一样看世界。

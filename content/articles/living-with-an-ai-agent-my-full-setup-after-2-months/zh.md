## 硬件

Mac Mini M4，16GB 内存。2026 年初买的。它放在我的客厅里，7x24 小时运行，为一切提供动力：AI Agent 网关、一个用于压缩的本地 LLM、偶尔的 Docker 容器，以及所有 18 个 cron 作业。

总磁盘占用：Agent 框架约 120MB + 本地 LLM 模型 6.6GB。

简而言之：这是一台 600 美元的电脑，只有三明治大小。它运行我的整个 AI 设置。没有云服务器或 GPU 集群，没有每月 AWS 账单。客厅里的一个盒子。

## Agent 框架：Hermes Agent

Hermes Agent 是 @NousResearch 的一个开源 Python 框架。它是一切的支柱。

界面是一个 Telegram 机器人。我像发短信给同事一样给它发消息。它可以访问终端、读写文件、搜索网络、获取完整网页、执行代码、运行 shell 脚本，以及使用我为它写的自定义技能。

它作为 macOS launchd 服务运行，意味着它在启动时启动，如果崩溃会自动重启。

Hermes 提供的是：

- 带有完整工具访问权限的 Telegram 机器人（终端、文件、网络搜索、网络获取、代码执行）
- cron 调度器 — 作业按计划运行，结果发送到 Telegram
- 技能系统 — 教 Agent 新能力的 markdown 文件
- 记忆系统 — Agent 跨会话读写持久文件
- 会话管理 — 压缩、空闲超时、每日重置
- MCP 支持 — 用于扩展工具访问的模型上下文协议服务器

它不提供的是：

- 多模型回退链（只有单一回退，如果两个模型都失败，作业就失败）
- 自动会话清理（需要手动脚本）
- 好的写作（模型限制，不是框架限制，后面会更多提到）

简而言之：把它想象成一个住在你的 Telegram 里的 AI 助手。但不仅仅是当你问问题时回答问题，它还在后台运行计划任务，记住对话之间的事情，可以使用工具，就像一个 24/7 工作、不睡觉、可以访问你的终端的初级员工。

## 模型栈

这里变得有趣了。我不使用一个模型。我用三个，每个做不同的工作。

**交互式聊天：通过 Z.AI 的 GLM-5**

当我直接给机器人发消息时，它使用 GLM-5，这是一个由智谱 AI 开发的中文模型，通过 Z.AI 的每月 21 美元编程计划访问。

为什么是 GLM-5？这是工具调用实际可用的最便宜模型。它推理良好，遵循复杂的多步指令，可靠地处理研究管道。然而，它的写作很差。每个草稿听起来都像 AI。但我不需要它写作，我需要它研究、执行和协调。写作我自己来处理。

**推理 token 陷阱：**

GLM-5 总是生成推理 token，在响应之前内部"思考"。hermes 有一个 reasoning_effort 配置选项应该控制这个。它对 GLM-5 什么都不做。模型无论如何都会思考。

那些推理 token 被存储在对话历史中。每个新消息都包含所有之前的推理。所以你的上下文很快填满。经过 15-20 次交换后，Agent 变慢了，因为它在重新处理对话中早期的数千个推理 token。

简而言之：想象一下，每次你问助手一个问题，他们都会在回答之前重新阅读从那天开始一整天的整个日记。

**Cron 作业：通过 Z.AI 的 GLM-4.7**

所有 18 个自动化作业都运行在 GLM-4.7 上，同一供应商，更便宜的模型。这很关键，因为 Z.AI 有一个速率限制：编程计划每 5 小时 600 次提示。如果 cron 作业使用 GLM-5，它们会消耗掉我的交互配额。

**压缩：通过本地 Ollama 的 Qwen3.5**

这是花了我最长的时间才弄对并教会我最多的东西。

每个 AI 模型都有一个上下文窗口，即它一次可以"看到"的文本量。随着对话变长，上下文填满。压缩总结旧消息以释放空间。

原来，压缩使用与其他所有东西相同的云 API（Z.AI）。这创建了一个死亡螺旋：

1. cron 作业运行 → 生成消息 → 上下文增长
2. 上下文达到阈值 → 触发压缩
3. 压缩调用 Z.AI → 使用 API 配额
4. 更多 cron 作业运行 → 更多压缩调用 → 更多配额使用
5. 速率限制击中 → 压缩静默失败 → 上下文继续增长
6. 上下文无限增长 → Agent 挂起 10+ 分钟试图处理臃肿的会话

我几天没注意到。Agent 只是……慢了。我以为是网络延迟。

修复：在 Mac Mini 上安装 Ollama，拉取 qwen3.5:4b（一个 3.4GB 模型，在 M4 芯片上以约 20 tokens/秒运行），并将压缩指向 localhost。完成。

**空闲超时发现：**

即使修复了压缩，Agent 在高峰时段仍然很慢。发现默认会话空闲超时有 1440 分钟 — 24 小时。意味着会话在白天从不重置。它们从早上 7 点早间简报贯穿每个 cron 作业直到第二天早上。

改为 60 分钟。现在会话在作业之间重置。响应时间从重负载期间 10+ 分钟变为 5 秒以下。

简而言之：想象你的桌子上一整天都堆满了文件，但你从不清理。到晚上，你找不到任何东西，每项任务都要花 10 倍时间，因为你在一堆旧笔记中翻找。空闲超时是你多久清理一次桌子。24 小时意味着你从不清理。60 分钟意味着每小时清理一次。同样工作完成，速度快 100 倍。

让配置工作的配置：

压缩阈值 0.50 意味着当上下文一半满时触发。Ollama 保持活跃 5 分钟意味着模型在空闲 5 分钟后自动从 RAM 卸载，为其他任务释放内存。

## 18 个 Cron 作业：自主研究管道

这是系统的核心。整天运行的 18 个计划作业，研究、监控、起草和维护基础设施。

每个作业运行 GLM-4.7。每个作业向 Telegram 交付结果。每个作业读写结构化上下文系统（稍后更多）。

**每日时间表**

07:00 — 早间简报。天气、加密货币价格、 stablecoin 锚定、RWA 板块动态、夜间研究回顾、热门 Hacker News 和 Reddit 帖子。读取我所有项目上下文文件以呈现今天重要的是什么。

07:30 — 竞争对手仪表板。追踪 11+ 个 CDP 和 stablecoin 协议、TVL 变化、治理提案、合作伙伴公告。我在 DeFi 工作。在竞争对手公开宣布之前知道他们在做什么就是这份工作。

09:00 和 17:00 — dune 监控。运行链上查询监控。一天两次。

10:00 — 魔法书起草。为我的 Telegram 频道起草视觉优先格式的帖子。

11:00 — 每日提醒。这个不一样。它读取我跨每个风险项目的所有项目上下文文件、Agent 本身、我的内容策略，并根据截止日期和跨项目紧张关系建议在哪里专注。这是一个优先级引擎。

12:00 — 草稿审查。挑选最好的未发布草稿，根据风格规则检查，发送到 Telegram 供我审查。

14:00 — 研究 AI。AI 新闻、论文、agent 框架。来源：arXiv、Reddit、Hacker News、Techmeme、新闻网站。

16:00 — 研究 arcana。删减。

18:00 — 研究 DeFi。stablecoin、RWA、治理、竞争对手分析。

20:00 — 研究深度探讨。从当天的发现中选择一个主题，深入分析。包含网络获取、数据编译和结构化报告。

21:00 — 健康检查。监控所有 cron、网关状态、磁盘空间、launchd 服务。

22:00 — 内容表现。追踪什么内容表现良好，记录指标。

23:00 — 夜间构建者。这是最狂野的。它查看研究期间识别的差距，如缺失脚本、破坏的数据源、需要的工具……并自主编写代码来填补。它们。它构建了监控脚本、修复了破坏的解析器、创建了新的数据管道。在我睡觉的时候。

**周期性时间表**

周一/周四 09:00 — 推广 CRM。自动化我的 BD 工作流程部分。

周一 09:00 — 每周情报。竞争情报简报。

周日/周三 20:00 — 学习摘要。一周策划的学习资源。

周日 08:00 — 每周计划者。读取表现数据、研究发现和项目状态 → 生成一周的内容和优先级计划。

**每小时**

每上午 9 点至晚上 8 点每小时 — 突发新闻。RSS 提要 + TVL 监控 + stablecoin 锚定警报 + 病毒式推文检测。如果有什么动静，我一小时内就知道。

现在，让我们了解一个研究 cron 实际上是如何工作的。

让我分解 research-ai 作业，因为它是最复杂的。提示约 3000 字。以下是它实际做的，步骤如下：

- 步骤 1 — 来源验证规则。在做任何其他事情之前，提示强调一条规则：永远不要包含你没有实际访问过 URL 的发现。如果你搜索但没有结果，就这么说。在包含任何发现之前，问问自己：我真的访问了这个 URL 吗？
这很重要，因为 GLM-4.7 会幻觉 URL。不经常，但足够了。这个指令将幻觉来源减少到接近零。

- 步骤 2 — 上下文加载。Agent 读取 7 个文件：它的持久记忆、操作日志、研究优先级、我的内容风格指南、过去草稿的风格修正、AI 研究档案和相关项目上下文文件。这意味着研究是根据我实际工作的事物为指导的。

- 步骤 3 — 邮件检查。使用 Himalaya（CLI 邮件客户端）检查机器人警报收件箱。捕获新闻通讯警报和自动通知。

- 步骤 4 — arXiv 扫描。运行自定义 Python 脚本查询 arXiv API 获取关于 AI agent、LLM 推理和工具使用的近期论文。按相关性过滤。

- 步骤 5 — 夜间构建者检查。通宵自主构建者有没有创建新工具？如果有，记下来。

- 步骤 6 — 网络搜索。需要多样化来源。提示明确说：先获取 Techmeme，然后 Hacker News，然后 Reddit，然后用网络搜索填补空白。永远不要仅仅依赖网络搜索。
这个来源多样性规则存在是因为 GLM-4.7 默认对一切使用 Reddit。Reddit 结果在搜索中排名很高。没有明确规则，每个早间简报都是 Reddit 独家的，持续好几天我才注意到。

- 步骤 7 — 深度阅读。网络获取 2-3 个最有趣的文章全文。

- 步骤 8 — 写作发现。这是强制性的。提示说："没有写作的会话就是失败。"发现被写入 AI 研究档案文件。这意味着明天会话从更丰富的知识库开始。

- 步骤 9 — 起草。如果有什么值得做内容的，用个人体验格式起草。先读风格修正。

- 步骤 10 — 质量检查。运行 check_draft.sh。必须根据风格规则得分 ≥70/100。

- 步骤 11 — 操作日志。一行总结发生了什么。

- 步骤 12 — 上下文更新。用会话结果更新项目上下文文件。

- 步骤 13 — 自我检查。"我是否向研究档案写了新内容？如果没有，回去做步骤 8。"这防止 Agent 做所有研究然后不保存任何东西。

简而言之：每个研究作业就像派一个员工去研究任务，附带 15 点清单。加载你的优先级。检查你的收件箱。扫描学术论文。阅读新闻。真的点击并阅读完整文章，不要只是浏览标题。写下你发现的东西。如果有趣，起草一些。检查你的草稿是否听起来像垃圾。记录你做了什么。更新团队你学到了什么。如果你没有写下任何东西，回去做它。

## 35 个 Shell 脚本：自动化层

每个脚本住在 ~/.hermes/scripts/，chmod 700，在 ~/scripts/ 做符号链接以便 Agent 可以轻松调用它们。

这些是系统的肌肉。Agent 是大脑。脚本是它与世界互动的方式。

**数据源脚本**

coingecko.sh — BTC/ETH/AAVE/MKR 价格，24 小时变化，stablecoin 锚定数据。不需要 API 密钥。

defillama.sh — CDP 协议 TVL 排名，stablecoin 市值。不需要 API 密钥。

rwa-tracker.sh — 完整 RWA 板块扫描。120+ 协议，$25B+ TVL。追踪顶级变动者。不需要 API 密钥。

hackernews.sh — 按关键词过滤的 Hacker News 热门故事（AI、DeFi、agents、LLM）。使用 Firebase API。不需要密钥。

reddit-digest.sh — 来自 r/LocalLLA、r/MachineLearning、r/defi 的热门帖子。使用 Reddit 的公共 JSON 端点。不需要密钥。

governance-tracker.sh — 我关心的协议的 Snapshot.org 活跃治理投票（Morpho、Curve、Lido）。加上论坛 RSS 提要。

arxiv-digest.py — 查询 arXiv API 获取关于 agents、推理、工具使用的最新论文。过滤和格式化。

fred.sh — 美联储经济数据。联邦利率、通胀、货币供应。免费 API 密钥。

**监控脚本**

tvl-monitor.sh — 缓存 TVL 快照，如果任何追踪的协议自上次检查变动 >10% 则警报。

stablecoin-supply-monitor.sh — 检测 >$100M USDT/USDC/DAI 铸造或销毁。大供应变化预示市场变动。

breaking-news.sh — 将所有监控器 + RSS 提要 + 病毒式推文检测结合到一个警报系统。

health-check.sh — 监控 cron 作业执行、launchd 服务、网关状态、磁盘空间、PAT 过期日期。

**内容管道脚本**

check_draft.sh — 根据我的风格规则对草稿评分 0-100。自动运行在每个草稿上。任何低于 70 的被标记。

draft-review.sh — 每天挑选最好的未发布草稿，标记格式错误的草稿。

mark_posted.sh — 将草稿移动到已发布存档，记录到内容日历。

log-performance.sh — 记录帖子指标：展示、互动、书签。

**基础设施脚本**

update-walnut.sh — 77 行 bash。在相关项目上下文文件开头添加时间戳日志条目。每个研究 cron 结束时调用。这是 cron 作业和上下文系统之间的粘合剂。

compact_memory.sh — 将会话日志归档超过 3 天的，操作日志修剪到 7 天。防止无限增长。

github-push-nightly.sh — 偏执脚本。在推送到 Git 之前运行 13 个正则表达式模式扫描泄露秘密（API 密钥、令牌、密码）。三层检查。

auto-update.sh — 每日 Hermes 框架更新触发器。

docker-cleanup.sh — 移除超过 4 小时的僵尸沙箱容器。

简而言之：脚本就像有一个工具箱。AI Agent 知道如何使用锤子、螺丝刀和扳手，但你必须给它实际的工具。每个脚本是 Agent 连接到特定数据源或能力的工具。没有它们，Agent 只能搜索网络。有了它们，它可以查询 DeFi 协议、扫描学术论文、监控链上活动、检查自己的健康状况，并推送到 GitHub。

这些都不复杂。大多数不到 50 行。价值在于让 35 个脚本在计划系统内协同工作。

## 6 个自定义技能：按需能力

Hermes 中的技能是 markdown 文件。你写一个 markdown 文档解释你希望 Agent 什么时候、怎样做，当被调用时 Agent 会读取它。

SKILL.md 文件是指令集。

1. walnuts — 跨项目上下文

在 Telegram 输入 walnuts → Agent 合成跨风险观点，包含优先级、障碍和紧张关系。

例子输出：

这是我的战略仪表板。在 Telegram 里输入一个词，10 秒，全面了解每个项目。

2. grimoire — Telegram 频道起草

为我的 Telegram 频道 @witcheergrimoire 起草视觉优先格式的帖子：1-4 句话 + 图片。包含 URL 验证规则，因为 GLM-5 约 5% 会幻觉 URL。

3. yari-intel — 竞争情报

按需市场分析：CDP TVL 排名、竞争对手深度探讨、RWA 机会映射、治理追踪。结合 DeFi llama 数据 + shell 脚本 + 网络搜索。

4. arcana-intel — 咨询市场研究

删减。

5. alpha-scanner — 跨来源信号检测

结合 Polymarket 赔率、stablecoin 流动、RWA 势头、社交信号（Reddit + Thread Reader）和治理 alpha。

6. voice-learn — 草稿反馈循环

这个值得有自己的部分（见下文）。简而言之：当我编辑一个 AI 草稿再发布时，这个技能提取我改变了什么和为什么，保存教训，每个未来草稿在写作之前读取那些教训。

简而言之：技能就像 AI 的 SOP（标准操作程序）。你写一个文档说"当我说 X，这样做 Y。"AI 在运行时读取文档并遵循它。你写一个 markdown 文件，Agent 就有新能力。

美丽的是迭代速度。写一个新技能需要 10 分钟。测试它需要一条 Telegram 消息。改进它需要编辑一个文本文件。与构建插件、部署服务、接入 API 相比。

## ALIVE：让一切复利积累的上下文系统

这是大多数人之前没有见过的部分，也是让最大差异的部分。

由 @stackwalnuts 创建，ALIVE 是一个结构化上下文系统。把它想象成磁盘上的个人知识图谱。基本单元是一个"walnut"，一个项目或领域的上下文容器。

**结构**

每个 walnut 有一个 _core/ 文件夹，包含 5 个文件：

- key.md — 身份、论点、与其它 walnuts 的连接
- now.md — 当前阶段、下一个动作、障碍
- tasks.md — 紧急 / 活跃 / 待办
- insights.md — 沉淀知识、学到的教训
- log.md — 会话历史（仅追加，最新在前）

就这些。

三个集成层让它完美工作：

**层 1：cron 作业写入 walnuts。**

每个研究 cron 结束时，update-walnut.sh 脚本将日志条目 prepend 到相关 walnut。research-ai 记录到 oz-agent walnut。research-defi 记录到 yari-finance walnut。research-arcana 记录到 arcana walnut。

这意味着每个 walnut 自动积累研究、发现和学习的历史。

**层 2：cron 作业从 walnuts 读取。**

在研究之前，每个 cron 读取相关 walnut 的任务和洞察。研究与我的实际工作对齐。

research-ai 读取 oz-agent 任务（"改进草稿质量"）→ 专门搜索提示工程技巧。

research-defi 读取 yari-finance 任务（"映射 RWA 合作伙伴机会"）→ 专门搜索 RWA 协议新闻。

早间简报读取所有 walnut now.md 文件 → 呈现今天跨每个项目重要的是什么。

**层 3：交互式 Telegram 获取 walnut 上下文。**

当我输入 walnuts，Agent 读取所有 5 个 walnut 文件，给我一个综合的跨风险视图。上下文保持在会话中直到空闲超时。

这就是它复利的方式：

1. 我更新 walnut 任务（手动或通过 Telegram）
2. cron 作业读取任务 → 研究与优先级对齐
3. 研究产生发现 → 发现更新 walnut 日志
4. 我根据发现调整任务
5. 下一个 cron 拾取调整的优先级
6. 研究变得更有针对性
7. 发现变得更相关
8. 重复

每个 cron 知道我要完成什么。它在搜索之前读取我的优先级。它在搜索之后写下它找到的东西。下一次会话从更聪明的基础开始。

Claude Code 和 Hermes Agent 本地读写相同的文件。相同的上下文层。

简而言之：想象你有 5 本笔记本，每个项目一本。在做研究之前，你重读相关笔记本，记住你正在做什么和你需要什么。研究之后，写下你发现的东西。明天，你再重读，你的研究更好，因为它建立在昨天的笔记之上。

ALIVE 就是这样，只是 AI Agent 自动做重读和记录，而且笔记本的结构方式让你的交互式 Agent 和自动化 cron 作业都可以使用。

## 教 AI 你的写作风格：风格反馈循环

这是我最兴奋的技术，而且是零成本。

**问题**

标准方法：写一个更好的提示。添加更多规则、更多例子、更多约束。"用个人体验格式写作。""使用小写。""不要用项目符号。""句子长度有变化。"

我有 39,000 字符的内容风格指令。模型仍然生产垃圾。更多规则 ≠ 更好输出。在某一点之后，模型无法整合更多抽象指令。

**修复：修正日志**

不是告诉 AI 如何写作，而是向它展示我改变了什么。

每个帖子之后，循环是：

1. 我在 Telegram 说"我调整了你的草稿"
2. voice-learn 技能激活
3. 它读取原始草稿和我的发布版本
4. 它提取每个差异：语气转变、删除、添加、词汇交换
5. 它将每个教训保存到 voice-corrections.md
6. 每个未来起草 cron 在写作之前读取 voice-corrections.md

**为什么修正胜过提示**

提示说"用个人体验格式写作。"

修正说："你写了'不是说这是 UST 2.0' — 我删除了它，因为先发制人的对冲削弱了专业性。不要再这样做了。"

151 行"当你做 X 时，我把它改成 Y，因为 Z。"

151 个具体例子。具体永远胜过抽象。每次。模型仍然生产 AI 垃圾，但它是更好的 AI 垃圾。而且修正会复利。每个通过反馈循环的草稿都添加更多例子。

模型不能像人一样写作。但它可以像一个稍微更好的 AI 写作，知道 151 件具体要避免的事情。当人类做最终编辑时，这就足够了。

简而言之：不是给人一本如何像你一样写作的书，而是每次他们写作时坐在他们旁边，指向具体句子，说"不要做这个，做这个。"28 次修正之后，他们开始自己避免错误。不完美。但足以让你的编辑时间从 80% 重写降到可能 50%。

## 我会做得不同的地方（两个月的教训）

1. 从第一天就使用本地压缩。云 API 压缩和速率限制创建静默失败。会话无限增长，你不会注意到，直到 Agent 每次响应挂起 10 分钟。本地 Ollama 压缩免费、可靠、零速率限制。

2. 立即设置每个作业模型配置。不要修补调度器代码。它在每次更新时被擦除。在 jobs.json 中每个作业配置设置 "model": "glm-4.7"。存活过更新，存活过补丁，永远不需要重新应用。

3. 不要信任 AI 写作。快速构建研究管道。从第一天就构建风格反馈循环。接受在几周内你将重写每个草稿的 50-80%。修正复利，但很慢。在期望可用草稿之前获得 50+ 修正条目。

4. 将会话空闲超时设置为 60 分钟。默认值通常更长。长会话 = 臃肿上下文 = 慢响应。短会话 = 新鲜上下文 = 快响应。研究仍然在会话之间保存到文件。

5. 在提示中强制来源多样性。如果你没有明确强制，模型会因为 Reddit 在搜索结果中排名高而对一切使用 Reddit。明确规则："先获取 Techmeme，然后 Hacker News，然后 Reddit，然后用网络搜索填补空白。"指定顺序。

6. 监控你的压缩管道。它静默失败。如果 Agent 突然需要 10 分钟响应并发送 8 条上下文转储消息，压缩可能坏了。检查模型、端点、manifest。ollama pull 强制干净下载。

## 哲学

这不是关于构建一个完美的 AI Agent。

GLM-5 生产垃圾。研究有时会幻觉 URL。cron 作业失败。压缩静默损坏。模型不能像人一样写作，可能永远都不会。

但它 7x24 小时运行。当我睡觉时，它扫描 arXiv 论文、监控 stablecoin 锚定、检查 Reddit 和 Hacker News、追踪竞争对手 TVL，并起草我早上编辑的帖子。当我醒来时，Telegram 里有一份夜间动态的简报和待审草稿队列。

Agent 是持久的。

而持久 × 时间 = 复利上下文。

经过数周运行，记忆文件已经捕获了模式。walnut 日志有历史。风格修正已经积累。每个新会话从比上一个更丰富的基础开始。研究更有针对性，因为它在搜索之前读取优先级。草稿稍微更好，因为它们在写作之前读取修正。优先级更敏锐，因为早间简报读取了昨天实际发生了什么。

这些都不神奇。它在围绕它积累的结构化上下文中。

walnuts、修正日志、研究档案。AI 是引擎。上下文是燃料。

而燃料会复利。

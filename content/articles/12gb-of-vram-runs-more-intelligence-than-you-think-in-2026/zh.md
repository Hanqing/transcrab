我知道，因为我测试过了。一块 12GB VRAM 的 RTX 3060。一个 90 亿参数的模型。零行手写代码。模型从一个提示写了整整一个太空射击游戏，13 个文件，3263 行代码。然后它自己迭代了 4 个阶段和 6 个提示，发现并修复了我从未指出的 bug。

这不是教程。这是我停止询问小模型是否足够好并开始测量后实际发生的事情。一切都开源了。提示词、配置、每次迭代。链接在最后。

## 实验

设置很简单。Qwen 3.5 9B 量化到 Q4_K_M，通过 llama.cpp 在 RTX 3060 上运行。基线 50 tokens/秒。在全上下文和工具调用开销的真实 agent 工作负载下 30-45。128K 上下文。256K 上下文时总 VRAM 使用量：8.2GB，剩余 4GB。

agent 工具是 Hermes Agent by NousResearch。31 个工具。文件操作、终端访问、代码执行。模型不只是生成文本。它读取文件、编写代码、运行命令并修复自己的输出。

我给了它一个提示：构建一个叫 Octopus Invaders 的太空射击游戏。完整规格。音频、像素画、物理、UI、多文件架构。一切一次完成。

第一个结果是黑屏。

## 翻转

90 亿参数模型输出黑屏不是失败。而是起点。

我打开浏览器。什么都没渲染。我打开控制台。到处是错误。我检查了模型写的每个文件并建立了 bug 列表。11 个具体问题。错误的变量作用域。缺失的函数调用。损坏的事件监听器。CSS 冲突。

我把那个列表反馈给同一 GPU 上的同一模型。同样的 9B。同样的 3060。我没有修改任何代码。只是精确的诊断。

模型在 11 个文件中修复了每一个 bug。游戏加载了。敌人出现了。子弹发射了。音频播放了。计分追踪了。从黑屏到可运行的游戏，只用了两个提示。

那一刻。那个翻转。这就是 12GB VRAM 在 2026 年能做的事，当你停止归咎于模型大小并开始改进提示词时。

## 迭代 3：自我调试的模型

我没有在可运行的游戏上停下来。我让同一个模型再跑了 4 个阶段。

阶段 0 是回归修复。模型在一个文件中重命名了一个变量但没有在另一个文件中重命名。敌人停止生成了。我准确地告诉它不匹配在哪里。它巧妙地修复了作用域问题。

阶段 1 是主页重新设计。游戏有两个相互冲突的启动系统。HTML 覆盖层和 canvas 按钮都想启动游戏。模型移除了 canvas 系统并统一了流程。

阶段 2 到 4 是游戏玩法优化。背景优化。关卡进度。子弹大小。总共 6 个提示。所有这些都是零手写代码。

但真正的发现来自迭代中期。模型发现了一个我从未提过的 bug。浏览器在提供缓存的静态文件。旧的 JavaScript 而不是新代码在运行。模型自己推理了这个问题，识别了缓存行为，并在每个脚本和样式表标签上添加了版本参数来强制重新加载。

一个 90 亿参数的模型在一块 250 美元的 GPU 上发现了一个浏览器缓存问题，诊断了根本原因并实施了修复。没有提示。没有暗示。它就是知道。

## 解锁一切的标志

大多数运行本地模型的人都在浪费性能。默认的 llama.cpp 设置给你更低速度的 4K 上下文。正确的标志给你 128K 上下文，同样速度，但工作空间大 32 倍。

这是确切的配置：

每个标志的作用：

-ngl 99 把每一层都放在 GPU 上。不做 CPU 卸载。如果放得下，就全部在显卡上运行。

-c 131072 设置 128K 上下文。这是模型在工作时可以把整个代码库放在内存中的地方。

-np 1 使用单个并行槽。节省 190MB VRAM。你是一个人跟一个模型对话。你不需要多个槽。

-fa on 启用 flash attention。这是关键的一个。没有它，生成速度会随着上下文填满而下降。有了它，4K 的 50 tok/s 在 512K 保持 50 tok/s。平坦曲线。没有惩罚。

--cache-type-k q4_0 --cache-type-v q4_0 量化 KV 缓存。这就是你怎么在 12GB 上装下 128K+ 上下文。没有它，缓存本身就会在你达到 32K 之前吃掉你的 VRAM。

默认设置和这个配置之间的区别是玩具和工具之间的区别。

## 工具：为什么 Hermes Agent 在小模型上胜出

模型是等式的一半。工具是另一半。

Hermes Agent by NousResearch 有 11 个模型特定的工具调用解析器。Qwen、DeepSeek、Llama、Mistral、GLM、Kimi 等等。当一个 9B 模型尝试调用工具时，解析器确切知道那个模型如何格式化输出。它捕获格式错误的 JSON。它处理部分完成。即使模型的输出很混乱，它也能正确路由调用。

这就是小模型在 Hermes 上能工作而在其他框架上失败的原因。解析器补偿了参数较少带来的粗糙边缘。9B 不像 70B 那样干净地格式化工具调用。Hermes 不介意。它解析模型给它的任何东西并使其工作。

我有两个 PR 合并到 Hermes 代码库中。两个都在提交同一天被 NousResearch 的创始人 Teknium 合并。

PR #1 修复了小模型的图像传递。14B 以下的模型不能使用较大模型使用的 MEDIA: 语法。我在网关中添加了自动文件路径检测，这样任何模型都可以通过 Telegram 原生传递图像，无需特殊格式。

PR #2 添加了本地模型名称和上下文长度的自动检测。在此修复之前，Hermes 对每个本地模型显示 "claude-opus-4.6" 和 "2M 上下文"。现在它在启动时查询你的 llama.cpp 服务器，显示真正的模型名称和实际分配的上下文。Teknium 将修复扩展到覆盖所有自定义端点。

另一个选择是 OpenClaw。超过 10 万行 TypeScript 代码。创始人于 2026 年 2 月加入 OpenAI。它依赖在本地推理时会中断的 provider API。我 DM 里的用户描述它每次更新都崩溃。设置繁琐。工具调用解析器是通用的，不是模型特定的。

这不是观点。这是代码层面的比较。Hermes 有 47 个工具和 11 个解析器。56000 行 Python。由独立研究实验室积极维护。对于消费级硬件上的本地模型，没有其他能与之相比。

## 数字：什么能装什么

这是你今天实际上能在消费级硬件上运行的东西。下面的每个数字都来自真实测试，不是规格表。

8GB VRAM（RTX 3060 8GB，4060，4060 Ti 8GB）

- 模型：Qwen 3.5 7B Q4_K_M（~4.5GB）
- 上下文：32K-64K，量化 KV 缓存
- 速度：~55-60 tok/s 估计
- 用例：编程助手、聊天、轻量工具调用

12GB VRAM（RTX 3060 12GB，4060 Ti 16GB）

- 模型：Qwen 3.5 9B Q4_K_M（5.3GB）
- 上下文：128K-512K，量化 KV 缓存
- 速度：50 tok/s，所有上下文长度平坦
- 256K 时 VRAM：8.2GB（4GB 余量）
- 用例：全自主 agent 与 Hermes，多文件代码生成

16GB VRAM（RTX 5070 Ti，4080）

- 模型：Qwen 3.5 14B Q4_K_M（~8.5GB）
- 上下文：128K+，还有空间
- 速度：~40-45 tok/s
- 用例：更强的推理、更好的工具调用、更复杂的架构

24GB VRAM（RTX 3090，4090）

- 模型：Qwen 3.5 27B Q4_K_M（16.7GB）
- 上下文：300K+
- 速度：3090 上 35 tok/s
- 用例：甜点。严肃的自主编程、研究、生产工作流

每个层级运行相同的标志。每个层级使用相同的工具。每个层级受益于相同的优化。唯一变量是有多少参数能装进你的 VRAM。

## 你的硬件足够了

每个云提供商的宣传都一样。你的 GPU 太小了。你的模型太弱了。为真正的服务付费。

我在一个 5 年前价值 250 美元的二手 GPU 上运行了一个 90 亿参数的模型。它在 13 个文件中写了 3263 行代码。它修复了自己的 bug。它在没有告诉的情况下发现了一个浏览器缓存问题。它以 50 tokens/秒的基线运行，在真实 agent 工作负载下 30-45，128K 上下文。

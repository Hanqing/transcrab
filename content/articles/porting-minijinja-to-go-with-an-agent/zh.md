写于 2026 年 1 月 14 日

事实证明，现在你可以直接移植项目了。今年夏天我已经尝试过这个实验，但当时时间有限，工作量有点大。不过，情况已经有所进展。昨天，我将 [MiniJinja](https://github.com/mitsuhiko/minijinja)（一个 Rust 实现的 Jinja2 模板引擎）移植到了原生 Go，而且我几乎完全是用一个 AI Agent 来完成这项工作的。实际上，除了在一些高层思路上给予指导外，我几乎没做什么。

我总共大概只花了 45 分钟主动参与。它在我监督下工作了大约 3 小时，然后又独自运行了 7 小时。这篇文章是对整个过程的回顾以及我从中得到的收获。

所有提示都是通过语音使用 [pi](https://buildwithpi.ai/) 完成的，一开始用的是 Opus 4.5，后来在修复长尾测试时切换到了 GPT-5.2 Codex。

## 什么是 MiniJinja

MiniJinja 是 Jinja2 的 Rust 重新实现。我最初写它是因为我想用 Rust 做一个基础设施自动化项目，而 Jinja 在这方面很流行。原始项目最终没有继续下去，但 MiniJinja 本身对我和其他用户来说一直很有用。

MiniJinja 的测试方式是快照测试：输入和预期输出，使用 [insta](https://insta.rs/) 来验证它们是否匹配。这些快照测试正是我想用来验证 Go 移植版本的。

## 测试驱动移植

我的初始提示要求 Agent 找出验证移植的方法。通过这次对话，我和 Agent 达成了一致路径：复用现有的 Rust 快照测试，并逐步移植（词法分析器 -> 解析器 -> 运行时）。

这意味着 Agent 在 Go 端构建了以下工具：

*   解析 Rust 的测试输入文件（这些文件将设置嵌入为 JSON 头部）。
*   解析参考的 insta `.snap` 快照并进行输出比较。
*   维护一个跳过列表，暂时跳过失败的测试。

这产生了一个相当不错的测试框架，反馈循环很紧密。Agent 有明确的目标（让所有测试通过）和进度规划（词法分析器 -> 解析器 -> 运行时）。紧密的反馈循环在最后阶段尤为重要，那时主要是把细节做对。每一个缺失的行为都会对应一个或多个失败的快照。

## 在 Pi 中分支

我使用 Pi 的分支功能会话来组织不同阶段。我回退到会话的较早部分，并使用分支切换功能自动告知 Agent 它已经完成了什么。这类似于压缩，但 Pi 会向我展示它放入上下文的内容。当 Pi 切换分支时，它会做两件事：

1.  它保持在同一个会话中，这样我可以四处浏览，但它会从较早的消息分出一个新分支。
2.  切换时，它会添加一个所做工作的摘要作为启动消息到分支点。我发现这很有帮助，可以避免 Agent 从头开始"视觉探索"来确定它已经走了多远。

如果不切换分支，我可能会创建新会话，并在周围留下更多计划文件，或者使用类似 Amp 的交接功能，这也允许 Agent 在需要更多信息时查阅早期对话。

## 首次出现分歧的迹象

有趣的是，Agent 很快从字面移植转向了行为移植。只要行为一致，我没有阻止它这样做。我让它这样做有几个原因。首先，代码库不算太大，所以我觉得如果需要可以在最后进行调整。让 Agent 继续它已经奏效的方法感觉是正确的策略。其次，这样更符合 Go 的惯用法。

例如，在运行时，它实现了一个树遍历解释器（不像 Rust 那样是字节码解释器），并且它决定使用 Go 的反射来实现值类型。我没有告诉它做这两件事中的任何一件，但它们比复制我的 Rust 解释器设计更有意义，后者部分是出于没有垃圾回收器或运行时类型信息的考虑。

## 我需要纠正的地方

另一方面，Agent 在让测试通过时做了一些我不同意的更改。它完全放弃了所有"必须失败"的测试，因为考虑到运行时差异，错误消息不可能完美复制。所以我不得不引导它采用模糊匹配。

它还想回退我想保留的行为（例如，精确的 HTML 转义语义，或者 `range` 必须返回迭代器）。我想如果我没有引导它，它可能无法在不走入歧途的情况下完成，或者我会对结果失去信心。

## 磨到全覆盖

一旦主要的语义不匹配问题得到解决，剩下的工作就是填补所有缺失的部分：缺失的过滤器和测试函数、循环额外功能、宏、调用块等。因为我想去睡觉了，我切换到 Codex 5.2 并排队了一些"如果测试还没通过就继续让所有测试通过"的提示，然后让它在压缩状态下工作。我有足够的信心，一旦基础部分完成，Agent 可以在没有指导的情况下让其余测试通过。

这个阶段在无监督的情况下通宵运行。

## 最终清理

在功能收敛之后，我要求 Agent 为内部函数添加文档并重新组织（比如将过滤器移到单独的文件中）。我还要求它像 Rust 代码库那样为所有函数和过滤器添加文档。这也是我设置 CI、发布流程的时候，并讨论了创建的内容，以提出一些最终润色建议，然后再合并。

## 临别思考

这里有几件事我觉得很有趣。

第一：这类移植现在成为可能。我知道移植在几个月前就已经可能了，但需要更多的关注。这改变了一些动态。我觉得技术选择不再受生态系统锁定的限制。当然，将 NumPy 移植到 Go 会更复杂，让它具有竞争力更是如此（里面有多年的优化）。但尽管如此，感觉现在可以使用更多的库了。

第二：对我来说，价值正在从代码转向测试和文档。一个好的测试套件实际上可能比代码更有价值。也就是说，这并不是主张将测试保密——生成具有良好覆盖率的测试也变得更容易了。然而，要让不同语言的代码库保持同步，你需要就共享测试达成一致，否则分歧是不可避免的。

最后，还有社会动态的问题。曾经，让别人把你的代码移植到其他语言是一件值得骄傲的事。这是成就的标志——一个项目"足够酷"，以至于有人愿意投入时间让它在其他地方可用。有了 Agent，它不会引发同样的感受。Will McGugan [也指出了这种变化](https://bsky.app/profile/willmcgugan.bsky.social/post/3mccn3l4qdk26)。

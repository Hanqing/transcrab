# 构建 Claude Code 的经验教训：提示缓存就是一切

工程界有句话说"缓存统治我周围的一切"，同样的规则也适用于 agent。

像 Claude Code 这样长时间运行的 agent 产品之所以可行，是因为提示缓存允许我们重用之前往返的计算，显著降低延迟和成本。

什么是提示缓存，它如何工作，你如何在技术上实现它？在 [@RLanceMartin 关于提示缓存和我们新的自动缓存发布的文章中阅读更多。](https://x.com/RLanceMartin/status/2024573404888911886)

在 Claude Code，我们围绕提示缓存构建整个工具。高提示缓存命中率降低成本，帮助我们为订阅计划创建更慷慨的速率限制，所以我们运行提示缓存命中率的警报，如果太低就宣布 SEV。

这些是我们从大规模优化提示缓存中学到的（通常反直觉的）经验。

## 为缓存布局你的提示

提示缓存通过前缀匹配工作——API 缓存从请求开始到每个 cache_control 断点的所有内容。这意味着你放置东西的顺序非常重要，你希望尽可能多的请求共享一个前缀。

最好的方式是静态内容在前，动态内容在后。对于 Claude Code，顺序如下：

1. 静态系统提示和工具（全局缓存）
2. Claude.MD（在项目内缓存）
3. 会话上下文（在会话内缓存）
4. 对话消息

这样我们最大化多少会话共享缓存命中。

但这可能出奇地脆弱！我们之前打破这个顺序的原因包括：在静态系统提示中放入深入的タイムスタンプ、非确定性打乱工具顺序定义、更新工具参数（例如 AgentTool 可以调用哪些 agent）等等。

## 使用消息进行更新

有时候你放入提示的信息会过时，例如如果你有 时间或用户更改了文件。更新提示可能很诱人，但这会导致缓存未命中，可能对用户来说相当昂贵。

考虑是否可以通过下一轮的消息传入这个信息。在 Claude Code，我们在下一条用户消息或工具结果中添加一个 `<system-reminder>` 标签，包含模型的更新信息（例如，现在是多伦多/周三），这有助于保留缓存。

## 会话中途不要更换模型

提示缓存是模型特有的，这可能使提示缓存的数学变得相当反直觉。

如果你在和 Opus 的对话中已经聊了 100k tokens，现在想问一个相当容易回答的问题，切换到 Haiku 实际上会比让 Opus 回答更贵，因为我们需要为 Haiku 重建提示缓存。

如果你需要切换模型，最好的方式是通过子 agent，Opus 可以在那里准备一个"交接"消息给另一个完成它需要完成的任务的模型。我们经常在 Claude Code 的 Explore agent 中这样做，它们使用 Haiku。

## 永远不要在会话中途添加或删除工具

在对话中更改工具集是人们最常见打破提示缓存的方式之一。这看起来很直觉——你应该只给模型你认为它现在需要的工具。但因为工具是缓存前缀的一部分，添加或删除工具会使整个对话的缓存失效。

计划模式——围绕缓存约束设计

计划模式是围绕缓存约束设计功能的一个很好的例子。直觉的方法是：当用户进入计划模式时，交换工具集，只包含只读工具。但这会破坏缓存。

相反，我们在所有请求中始终保留所有工具，并将 EnterPlanMode 和 ExitPlanMode 作为工具本身。当用户打开计划模式时，agent 收到一条系统消息，解释它在计划模式下以及指令是什么——探索代码库，不要编辑文件，计划完成时调用 ExitPlanMode。工具定义从不改变。

这有一个额外的好处：因为 EnterPlanMode 是一个模型可以调用的工具，当它检测到一个难题时，它可以自主进入计划模式，而不会破坏任何缓存。

工具搜索——延迟加载而不是删除

同样的原则适用于我们的工具搜索功能。Claude Code 可以加载数十个 MCP 工具，将它们全部包含在每个请求中会很昂贵。但在对话中删除它们会破坏缓存。

我们的解决方案：defer_loading。我们不删除工具，而是发送轻量级存根——只是工具名称，带有 defer_loading: true——模型可以通过 ToolSearch 工具在需要时"发现"它们。只有当模型选择它们时才加载完整的工具 schema。这保持缓存前缀稳定：相同的存根始终以相同顺序存在。

幸运的是，你可以使用我们的 API 中的[工具搜索](https://platform.claude.com/docs/en/agents-and-tools/tool-use/tool-search-tool)工具来简化这个。

## 分叉上下文——压缩

压缩发生在你用完上下文窗口时。我们总结迄今为止的对话，并在新的会话中继续这个摘要。

令人惊讶的是，压缩在提示缓存方面有很多可能反直觉的边缘情况。

特别是，当我们压缩时，需要将整个对话发送给模型生成摘要。如果这是一个单独的 API 调用，有不同的系统提示且没有工具（这是简单的实现），主对话的缓存前缀根本不匹配。你需要为所有这些输入 tokens 支付全价，大大增加用户的成本。

解决方案——缓存安全分叉

当我们运行压缩时，我们使用与父对话完全相同的系统提示、用户上下文、系统上下文和工具定义。我们在前面加上父对话的消息，然后在末尾追加压缩提示作为新的用户消息。

从 API 的角度来看，这个请求看起来几乎与父请求的最后一次请求相同——相同的前缀、相同的工具、相同的历史——所以缓存前缀被重用。唯一的新 tokens 是压缩提示本身。

但这确实意味着我们需要保存一个"压缩缓冲区"，以便在上下文窗口中有足够的空间来包含压缩消息和摘要输出 tokens。

压缩很棘手，但幸运的是，你不需要自己学习这些经验——基于我们从 Claude Code 学到的，我们直接[将压缩](https://platform.claude.com/docs/en/build-with-claude/compaction#prompt-caching)构建到 API 中，所以你可以在自己的应用中应用这些模式。

## 经验教训

1. 提示缓存是前缀匹配。前缀中任何地方的任何更改都会使之后的所有内容失效。围绕这个约束设计你的整个系统。把顺序做对，大部分缓存就会自动工作。

2. 使用消息而不是系统提示更改。你可能很想编辑系统提示来做进入计划模式、更改日期等事情，但实际上最好在对话中把这些插入消息里。

3. 不要在对话中途更改工具或模型。使用工具来建模状态转换（如计划模式），而不是更改工具集。延迟加载工具而不是删除工具。

4. 像监控正常运行时间一样监控你的缓存命中率。我们对缓存破坏发出警报并将其视为事件。几个百分点的缓存未命中率可能会显著影响成本和延迟。

5. 分叉操作需要共享父级的前缀。如果你需要运行辅助计算（压缩、摘要、技能执行），使用相同的缓存安全参数，以便在父级前缀上获得缓存命中。

Claude Code 从第一天起就围绕提示缓存构建，如果你正在构建 agent，你也应该这样做。
